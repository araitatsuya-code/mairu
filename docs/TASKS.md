# Mairu タスクリスト

設計ドキュメント（`docs/gmail_cleaner_design_v2.md`）を実装へ落とし込むための作業一覧です。2026年3月7日時点で `MAIRU-012 / #12` まで完了しており、次の主作業は `MAIRU-013 / #13` です。

実際の着手は、このタスクリストを分割した issue 単位の一覧である `docs/ISSUES.md` を基準に進めます。

## このタスクリストの使い方
- フェーズ番号は設計書の「開発フェーズ」と揃え、着手順の混乱を防ぐ。
- `Phase 1` から `Phase 4` をコア機能の完成範囲とし、ここまでを最優先で進める。
- `Phase 5` と `Phase 6` は拡張機能と配布整備として扱う。
- AIアシスタント機能は設計書どおり v2 以降のバックログに分離し、コア機能完了前には着手しない。
- 各フェーズは「主要タスク」と「完了条件」の両方を満たしたら完了とする。

## 最優先範囲（最初のリリース候補）

## Phase 1: Wails プロジェクト作成 + OAuth 認証 + Gmail API 接続確認

### 目的
ローカルデスクトップアプリとして起動し、Google ログイン後に Gmail API へ安全に接続できる最小構成を完成させる。

### 主要タスク
- [x] `app.go` に最小限の公開メソッドを定義し、Wails の Go バインディング生成が通ることを確認する。
- [x] `frontend/src/pages/Settings` の雛形を作り、認証状態と必須設定の有無を表示できるようにする。
- [x] Google OAuth 2.0 の PKCE + localhost リダイレクトフローを実装する。
- [x] Google Cloud Console の設定手順（デスクトップアプリ、必要スコープ）をドキュメントに反映する。
- [ ] 必要スコープ `gmail.modify` `gmail.labels` `gmail.metadata` をコード上で明示する。
- [x] OAuth トークンの保存先を OS キーチェーンに統一し、平文保存を避ける。
- [x] 認証済みトークンを使って Gmail API クライアントを初期化できるようにする。
- [x] Gmail のプロフィール取得または少量のメール一覧取得で接続確認を行う。
- [x] 認証エラー、トークン失効、再ログイン要求のエラー経路を定義する。

### 完了条件

- [x] ログイン後、Gmail API からユーザー情報またはメール一覧を取得できる。
- [x] 再起動後も保存済みトークンで再認証なしに接続できる。

## Phase 2: Claude API でメール分類 + 承認画面 UI

### 目的
取得したメールを Claude で分類し、信頼度に応じて自動実行またはユーザー承認へ振り分ける。

### 主要タスク
- [x] Claude API クライアントを実装し、モデル名を設定値として切り替えられるようにする。
- [x] Claude API キーを `Settings` 画面から登録できるようにし、保存先は OS キーチェーンに統一する。
- [ ] 送信者、件名、冒頭 200 文字のみを送る分類プロンプトを実装する。
- [x] 分類カテゴリ `important` `newsletter` `junk` `archive` `unread_priority` を DTO に定義する。
- [ ] 最大 50 件のバッチ分類を実装し、入力件数が多い場合は分割実行する。
- [x] `confidence` に応じた処理分岐（自動実行、要承認、理由補足、保留）を実装する。
- [x] `frontend/src/pages/Classify` に分類結果確認画面を実装する。
- [x] 分類理由、信頼度、推奨アクションを UI に表示する。
- [x] 重要ラベル、要確認ラベルなど必要な Gmail ラベルを作成・適用できるようにする。
- [x] 削除や一括処理は必ず確認ステップを挟むように UI / バックエンドの境界を定義する。
- [x] 承認済みアクションを Gmail API の削除、アーカイブ、ラベル付与へ接続する。
- [ ] 分類件数と概算トークン数を記録し、コスト感を把握できるようにする。

### 完了条件
- [x] Gmail から取得したメール 50 件を分類し、結果を UI に表示できる。
- [x] 信頼度ルールに従って自動実行と承認待ちが分かれる。
- [x] ユーザー承認後に Gmail 側のアクションが正しく反映される。

## Phase 4: 定期自動実行 + OS 通知

### 目的
アプリ起動中にバックグラウンドで定期処理を実行し、ユーザーへ結果を通知できるようにする。

### 主要タスク
- [ ] Go の goroutine で動くスケジューラー基盤を実装する。
- [ ] 新着メール分類を 1 日 1 回（設定変更可）で実行できるようにする。
- [ ] ブロックリスト更新を 1 日 1 回実行できるようにする。
- [ ] 既知ブロック送信者の即時処理を 30 分ごとに実行できるようにする。
- [ ] 同時実行による二重処理を防ぐロックまたはジョブ状態管理を実装する。
- [ ] 処理開始、成功、一部失敗をログへ残せるようにする。
- [ ] 一時ネットワークエラー時は最大 3 回まで自動リトライする。
- [ ] Gmail / Claude のレートリミットに備え、指数バックオフを実装する。
- [ ] 認証エラーや API キー不正時は設定画面へ誘導できる通知導線を用意する。
- [ ] OS 標準通知で完了件数と承認待ち件数を表示する。
- [ ] `Settings` 画面にスケジュール設定と通知 ON/OFF の UI を追加する。

### 完了条件
- [ ] アプリ起動中に定期ジョブが設定どおり動く。
- [ ] 成功 / 一部失敗 / 承認待ちが OS 通知で区別して伝わる。
- [ ] 同一ジョブの重複実行や無限リトライが発生しない。
