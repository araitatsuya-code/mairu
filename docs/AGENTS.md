# AGENTS.md

Codexエージェントがこのリポジトリで作業するときの運用ルールをまとめています。`docs/DEVELOPMENT.md` と併せて参照してください。

## 1. 目的
- ユーザーとのやり取りをスムーズにし、セッションを跨いでも一貫した対応ができるようにする。
- 手戻りを減らすため、コマンド利用ポリシーや禁則事項を明文化する。

## 2. 原則
1. **サマリーや新規ドキュメントは日本語で作成**する（README/開発ガイド/設計資料なども含む）。
2. ユーザーの既存変更は上書きしない。汚れたワークツリーを見つけたら指示を仰ぐ。
3. `docs/gmail_cleaner_design_v2.md` を常に最新の仕様ソースとして扱う。
4. 作業内容が複雑な場合は Codex CLI の `/plan` を使い、完了ごとに更新する。

## 3. コマンド利用
- 基本は `[
- 基本は `"bash", "-lc", "<command>"` 形式で実行し、`workdir` を必ず指定する。
- 検索は `rg` / `rg --files` を優先。なければ代替を検討。
- 書き込みやGit操作でサンドボックスに阻まれたら、`with_escalated_permissions` を付けて再実行する。
- ネットワークアクセスが必要な場合は sandbox 設定を確認し、許可されていないときは代替案を提示する。

## 4. ファイル編集ポリシー
- ASCIIを基本とし、既存でUnicode絵文字を使用している箇所のみ例外を認める。
- コメントは最小限。コードが自明でない場合のみ補足を書く。
- 既存の未コミット変更を削除・上書きしない。必要ならユーザーへ確認する。

## 5. ドキュメント更新
- 仕様/構成が変わったら `docs/gmail_cleaner_design_v2.md` か `README.md`、必要に応じて `docs/DEVELOPMENT.md` を更新する。
- エージェント向けルールを変える場合はこの `docs/AGENTS.md` へ追記し、変更理由をコミットメッセージに含める。

## 6. コミット・プッシュ
1. `git status -sb` で対象ファイルのみが変更されていることを確認。
2. `git add ...` → `git commit -m "<type>: <summary>"` 形式でコミット。
3. `git push` まで行い、結果を最終メッセージに記載。

## 7. 応答フォーマット
- 最終回答は簡潔な日本語の箇条書き。
- ファイル参照は `path/to/file:line` 形式、行範囲は使わない。
- 追加でやるべきことが明確な場合のみ「次のステップ」を提案する。

## 8. トラブルシュート
- 予期しないファイルが変更されていたら **即座に作業を止め、ユーザーへ報告**。
- コマンドが sandbox 制限で失敗したときは、理由を示しつつ再実行可否を判断。
- どうしても作業継続が困難な場合は、状況と提案（例: 手動操作、追加情報）をまとめてユーザーに伝える。
