# MAIRU-018 要件定義

大量メール耐性と段階実行

最小追加で実用に持っていく方針

## 1. 目的
- 件数が多い inbox でも、Gmail API / Claude API の制限超過や長時間停止を避けながら処理を継続できる実行要件を定義する。
- 既存の `scheduler` `settings` `action_logs` を活用し、大幅な再設計なしで「新着の安定処理」と「backlog の段階消化」を成立させる。
- 破壊的操作を自動化しすぎず、安全に日常利用できる運用境界を明確にする。

## 2. スコープ
- スコープ内:
  - 実行モード要件（`dry-run` `safe-run` `full-run`）
  - 対象絞り込み要件（新着処理、手動 backlog 処理）
  - 50 件バッチ分割、タイムアウト、再試行
  - バッチ単位チェックポイントと中断/再開
  - 既存ログを使った重複実行抑止
  - 要約通知と最低限の観測指標
- スコープ外:
  - inbox 全件を自動探索し切る仕組み
  - ドメイン連鎖による自動探索拡大
  - 独立した watchdog 常駐プロセス
  - Claude 分類精度そのものの改善
  - UI デザインの最終調整
  - v2+ のチャット機能

## 3. 実装方針
- 既存の `scheduler` をそのまま使い、定期実行・重複起動防止・指数バックオフを流用する。
- 中断/再開の checkpoint はまず `settings` に保存し、専用ジョブテーブルの追加は後回しにする。
- Gmail 反映済み判定は `action_logs` を利用し、`message_id` と実行アクションを手がかりに重複適用を避ける。
- v1 では「1 回で大量 backlog を完走する」ことよりも、「途中停止しても安全に再開できる」ことを優先する。

## 4. 運用前提
- Claude 分類は `50 件` 単位を維持する。
- 定期実行は backlog 全量ではなく「前回成功時刻以降の新着」を対象にする。
- backlog 消化は手動実行で行い、1 回の対象上限は `500 件` を標準、`1,000 件` を上限目安にする。
- checkpoint は `50 件バッチ完了ごと` に更新する。
- 自動実行と再開実行は既定で `safe-run` とする。
- `full-run` は UI で明示承認された手動実行のみに限定する。

## 5. 機能要件

### FR-01 実行モード
- `dry-run`: 分類結果と推奨アクションのみ記録し、Gmail 変更は行わない。
- `safe-run`: 削除を禁止し、ラベル付与・既読化・アーカイブのみ許可する。
- `full-run`: 明示承認済みの削除を含めて実行する。
- 定期実行と再開実行で許可するモードは `safe-run` のみとする。

### FR-02 対象絞り込み
- 定期実行は `last_run_at` 以降に到着したメールのみを対象にする。
- 初回起動や `last_run_at` 未設定時は、定期実行では大量 backlog を自動処理せず、手動実行へ誘導する。
- 手動 backlog 実行では、ユーザーが指定した上限件数までを対象に処理する。
- v1 ではドメイン単位の自動拡大は行わない。

### FR-03 50 件バッチ処理
- 対象メールは `50 件` ごとに分類バッチへ分割し、順次処理する。
- 各バッチは「抽出 -> Claude 分類 -> 承認判定 -> Gmail 反映」の順で完結させる。
- 1 バッチ完了ごとに checkpoint を永続化する。
- バッチ途中で失敗した場合は、そのバッチを未完了として扱い、次回は同じバッチから再開する。

### FR-04 チェックポイント再開
- 最低限、以下を `settings` に永続化する:
  - 実行種別（定期実行 / 手動 backlog）
  - 実行モード
  - 抽出条件（`last_run_at` または手動指定件数）
  - 最後に完了したバッチ番号
  - 成功 / 失敗 / 承認待ち / スキップ件数
  - 最終更新時刻
- 再開時は checkpoint の抽出条件で対象を再取得し、完了済みバッチ数ぶんをスキップして続行する。
- Gmail 反映前に `action_logs` を参照し、同一 `message_id` に同一アクションが `success` 済みであれば再適用しない。

### FR-05 API 制限回避
- Gmail / Claude 呼び出しは既存 scheduler の指数バックオフを流用し、一時エラーは最大 `3 回` まで再試行する。
- `429` `5xx` `timeout` など一時エラーは再試行対象として扱う。
- 再試行上限を超えた場合はジョブ全体を `interrupted` として終了し、次回は最後の完了バッチから再開する。
- 認証エラーや設定不備は再試行しないエラーとし、即時停止してユーザーへ通知する。

### FR-06 タイムアウト
- API 呼び出しごとにタイムアウトを設定する。
- 1 バッチ全体にもタイムアウトを設定する。
- v1 では独立 watchdog は導入せず、`context.WithTimeout` ベースの停止制御を優先する。

### FR-07 安全ガード
- 定期実行と再開実行では削除を実行しない。
- `full-run` では 1 ジョブあたりの削除件数上限を設定可能にする。
- `full-run` では削除比率が閾値を超える場合、自動停止して承認待ちに切り替える。
- `important` 相当カテゴリの自動削除は禁止する。

### FR-08 探索拡大の最小実装
- v1 の自動探索拡大は「同一送信者メールアドレスの完全一致候補表示」までに留める。
- 完全一致候補は自動投入せず、手動 backlog の次回候補として提示する。
- ドメイン単位の展開、優先度キュー、連鎖探索は後続 issue に送る。

### FR-09 観測と通知
- 最低限、以下を記録する:
  - 処理件数
  - 成功 / 失敗 / 承認待ち / スキップ件数
  - バッチ番号
  - 再試行回数
  - 停止理由（`timeout` `retry_exhausted` `auth_error` `config_error` `canceled`）
- ジョブ完了 / 一部失敗 / 停止時に要約通知を出す。
- `last_run_at` はジョブ成功時のみ更新する。

## 6. 非機能要件
- NFR-01 安定性: 単一バッチ失敗でデータ破損せず、次回再開できる。
- NFR-02 再開性: アプリ再起動後でも最後の完了バッチから再開できる。
- NFR-03 安全性: 定期実行で削除が走らない。
- NFR-04 可観測性: ログと通知だけで停止理由を追跡できる。
- NFR-05 実装容易性: 新規の大規模ジョブキュー基盤を追加しない。

## 7. 初期パラメータ案
- `manual_backlog_size_default`: 500
- `manual_backlog_size_max`: 1000
- `classify_batch_size`: 50
- `max_retry_per_batch`: 3
- `api_timeout_seconds`: 30
- `batch_timeout_seconds`: 120
- `delete_limit_per_job`: 100
- `delete_ratio_guard`: 0.15

## 8. 受け入れ条件
- AC-01 新着 100 件: 定期 `safe-run` で完了し、統計と `last_run_at` が更新される。
- AC-02 手動 backlog 1,000 件: 途中中断後に再開し、最後の完了バッチから続行できる。
- AC-03 重複防止: 再開時に `action_logs` 上で `success` 済みの Gmail アクションは二重適用されない。
- AC-04 API 異常系: `429` `5xx` `timeout` 発生時に再試行後、上限超過なら停止理由を残して終了する。
- AC-05 安全性: 定期実行では削除が一切走らず、`full-run` は手動承認時のみ実行できる。
- AC-06 大量 backlog: `14,000 件` の inbox でも 1 回で全量処理を要求せず、複数回の手動実行で継続消化できる。

## 9. 後続 issue 分割案
- 定期 `safe-run` の対象絞り込み実装（`last_run_at` ベース）
- 50 件バッチ checkpoint の保存 / 再開
- `action_logs` を使った Gmail アクション重複防止
- API timeout と再試行エラー分類の整理
- 手動 backlog 実行の件数上限と再開導線
- 同一送信者メールアドレス完全一致候補の表示
- ジョブ停止理由の記録と要約通知の強化

## 10. 将来拡張として扱う項目
- ドメイン単位の探索拡大
- 優先度付き探索キュー
- heartbeat / watchdog による `stalled` 検知
- 専用ジョブテーブルによる厳密な対象スナップショット管理
